# Typed Path Guide

`@PersistentModel` generates a typed key/path mapping layer that helps Core Data code survive
property renames without falling back to fragile string literals.

This guide focuses on one specific feature:

- how `Keys`, `path`, and `__cdFieldTable` support `NSSortDescriptor`
- how the same mapping supports `%K`-based `NSPredicate`
- when this feature solves a real problem
- when you can safely ignore it

It does not try to document every generated member on `@PersistentModel`; it focuses only on the
typed key/path surface relevant to sort and predicate construction.

## Why This Exists

Core Data and SwiftData both have a practical long-term pain point:

- after an app ships
- after a model starts syncing across devices or servers
- after lightweight migration becomes the only safe migration path

renaming a persisted field becomes expensive.

Sometimes the schema name is no longer ideal:

- the original name was too vague
- the name reflected an old product term
- the Swift-facing API became awkward

The usual workaround is to keep the persisted field name unchanged and expose a better Swift name
through mapping.

That is manageable for reading and writing values, but it creates a second problem:

- the mapped Swift property is often a computed or remapped layer
- computed properties cannot be used directly in `NSSortDescriptor`
- computed properties cannot be used directly in store-backed `NSPredicate`

This guide's feature exists to solve exactly that gap.

## The Core Idea

You write a Swift-facing property name that makes sense in application code:

```swift
@Attribute(persistentName: "name")
var title: String = ""
```

Here:

- Swift name: `title`
- persisted Core Data field name: `name`

The macro then generates:

- `Keys`
- `Paths`
- `path`
- `__cdFieldTable`

That generated metadata lets the library map:

- `title`
- `path.title`

back to:

- `name`

when building sort and predicate strings.

## When You Should Use It

Use this feature when at least one of these is true:

- your persisted field names are already shipped and hard to migrate
- your store schema must stay lightweight-migration-friendly
- you want better Swift-facing names than the database names
- you use composition or relationship subpaths and want one consistent mapping surface
- you want to eliminate hand-written key strings from sort and predicate code

## When You Can Ignore It

If your model does **not** have this problem, you do not need to lean on this feature.

If:

- your Swift property names already match the persisted field names
- you do not need remapped properties
- you are comfortable writing direct Core Data key strings in a small codebase

then this layer is optional.

It is not a mandatory style rule.

## Performance Note

This mapping layer is generated by macros, and key translation happens by resolving already
generated metadata at runtime.

That does **not** introduce store-side performance cost:

- Core Data still receives the final key path string it expects
- the mapping work happens before the fetch/sort is handed to Core Data
- no reflection is involved

So the feature improves API stability and readability, not query semantics.

## Flat Keys

For flat fields, `Keys` is the smallest surface:

```swift
let descriptor = NSSortDescriptor(Item.self, key: .date, ascending: true)
```

If the Swift property is:

```swift
@Attribute(persistentName: "timestamp")
var date: Date? = nil
```

then the generated sort descriptor still uses:

```swift
"timestamp"
```

instead of `"date"`.

That mapping is automatic.

## Typed Paths

`path` is the more flexible surface.

It supports:

- normal attributes
- renamed attributes
- to-one relationship subpaths
- composition subpaths
- to-many predicate quantifiers

Example:

```swift
let sort = try NSSortDescriptor(
  Item.self,
  path: Item.path.title,
  order: .asc
)
```

If `title` is stored as `"name"`, the descriptor still sorts by the persisted field name.

### To-one Relationship Predicates

The macro also exposes the relationship itself through a `CDToOneRelationPath`.
On that path you can now call:

```swift
Item.path.category.equals(categoryObject)
Item.path.category.isNil()
Item.path.category.isNotNil()
```

`equals(_:)` compares the relationship column to the actual Core Data object in the
current context, which avoids joining through child attributes when the object already
exists. Use `isNil()`/`isNotNil()` when you need to test for the absence or presence of
the relationship without fetching a target.

### Composition Paths

If you have:

```swift
@Composition
struct GeoPoint {
  @CompositionField(persistentName: "lat")
  var latitude: Double = 0

  @CompositionField(persistentName: "lng")
  var longitude: Double = 0
}

@objc(Item)
@PersistentModel
final class Item: NSManagedObject {
  @Attribute(storageMethod: .composition)
  var location: GeoPoint? = nil
}
```

you can write:

```swift
let sort = try NSSortDescriptor(
  Item.self,
  path: Item.path.location.latitude,
  order: .desc
)
```

The macro-generated metadata resolves that to the correct persistent field path.

So even though source code uses:

- `location.latitude`

the generated path can still resolve to a persistent field path like:

- `location.lat`

### Relationship Paths

To-one relationship subpaths also work:

```swift
@objc(Category)
@PersistentModel
final class Category: NSManagedObject {
  @Attribute(persistentName: "display_name")
  var name: String = ""
}

@objc(Item)
@PersistentModel
final class Item: NSManagedObject {
  @Relationship(
    persistentName: "primary_category",
    inverse: "items",
    deleteRule: .nullify
  )
  var category: Category?
}

let sort = try NSSortDescriptor(
  Item.self,
  path: Item.path.category.name,
  order: .asc
)
```

Current rule:

- to-one relationship paths are supported for sorting
- to-many relationship paths are **not** supported for sorting

This means:

```swift
Item.path.tags.name
```

is fine for predicate building through quantifiers, but not for `NSSortDescriptor`.

With the example above, the source path:

- `Item.path.category.name`

can resolve internally to a persistent key path like:

- `primary_category.display_name`

## `%K`-Based NSPredicate

The same mapping surface also helps with store-backed predicate construction.

You can write raw `%K` predicates without hand-maintaining schema strings:

```swift
let predicate = NSPredicate(
  format: "%K == %@",
  Item.path.title.raw,
  "hello"
)
```

If `title` maps to persisted field `name`, then:

- `Item.path.title.raw`

produces:

- `"name"`

This is one of the main reasons the feature exists.

Without it, renaming the Swift-facing property while keeping the persisted field stable would force
you back to manual string management.

## Predicate Helper Methods

You can also build simple predicates through the typed path helpers:

```swift
let predicate = Item.path.title.equals("hello")
```

Other examples:

```swift
let predicate = Item.path.title.contains("Core Data")
let predicate = Item.path.score.greaterThan(80)
let predicate = Item.path.createdAt.lessThanOrEqual(Date())
```

These helpers still compile down to `%K`-style `NSPredicate` construction using the mapped
persistent path.

For `.raw` storage, `equals` and `notEquals` also accept the `RawRepresentable` value directly:

```swift
let predicate = Item.path.status.equals(Status.published)
```

The helper converts the enum to its `rawValue` before building the predicate.

Current boundary:

- `.raw` supports `equals` and `notEquals` with `RawRepresentable` values
- `.codable` and `.transformed` do not expose store-backed typed predicate helpers

## To-Many Quantifiers

For to-many relationships, use:

- `.any`
- `.all`
- `.none`

Example:

```swift
let anySwiftTag = Item.path.tags.any.name.equals("Swift")
let allHighScore = Item.path.tags.all.score.greaterThan(80)
let noLegacyTag = Item.path.tags.none.name.contains("legacy")
```

These are translated to Core Data-compatible `NSPredicate` forms.

The important part is that renamed fields still map correctly through the same table.

## Current Boundaries

Currently:

- flat key sort is supported
- to-one relationship path sort is supported
- composition leaf sort is supported
- to-many relationship sort is not supported
- `%K` predicate building is the recommended query path
- helper methods on typed paths build `NSPredicate`, not a separate custom predicate type

## Takeaway

This feature is not about making Core Data queries more powerful than before.

It is about making them more maintainable after your model has already shipped.

If you have never needed to rename a persisted field while keeping lightweight migration safe, you
may not need it.

If you have, this is the layer that lets you keep a clean Swift API **and** still use the field in
`NSSortDescriptor` and store-backed `NSPredicate`.
